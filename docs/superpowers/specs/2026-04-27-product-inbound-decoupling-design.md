# 商品上架与入库登记解耦设计方案

## 1. 背景与目标
当前系统中，“商品入库”与“上架”逻辑绑定较深。用户希望将两者解耦：
- **上架（Listing）**：属于销售/档案管理范畴，负责建立商品的基本信息。
- **入库（Inbound）**：属于物流/仓储范畴，负责增量更新库存，且支持为“未上架”商品做入库记录。

## 2. 核心逻辑变更：双轨制
我们将业务拆分为两个独立流转的过程：

### A. 商品档案管理 (上架/维护)
- **入口**：库存看板 -> [➕ 商品上架]（由原“商品入库登记”更名）。
- **职责**：定义商品的静态属性（条码、名称、图片、卖点、标准售价）。
- **特性**：只有在此处“上架”后的商品，才会出现在库存看板列表中。

### B. 智能入库登记工作台 (新增模块)
- **入口**：左侧导航栏 -> [📥 商品入库登记]。
- **交互流程**：
    1. **条码触发**：店员扫描或录入条码。
    2. **自动识别**：
        - **已上架**：自动从 `ProductDataManager` 读取信息并填充预览卡片，店员仅录入数量、成本、供应商。
        - **未上架**：表单留白，店员手动输入。信息仅保存在“入库记录”中，**不自动上架**。
    3. **状态提醒**：对于未上架的入库商品，在记录中标记为“待上架”，提醒店员后续处理。

## 3. 详细组件设计

### 3.1 独立模块：`InboundModule` [NEW]
- **布局**：采用左右 Master-Detail 结构（与商品模块保持一致）。
- **Master 区域**：
    - 顶部：醒目的条码输入框（带自动聚焦）。
    - 中间：动态展示匹配到的商品预览卡片（图片、名称、当前库存）。
    - 底部：录入表单（数量、进价、生产日期、供应商）。
- **Detail 区域**：
    - 展示选中的历史入库记录详情。

### 3.2 现有模块修改：`ProductModule` [MODIFY]
- **功能去重**：移除原本的“入库记录回溯”标签页。
- **UI 调整**：将主按钮更名为“商品上架”。
- **引导提示**：增加“从入库记录中提取”功能，方便将已入库但未上架的商品快速转为档案。

### 3.3 数据管理：`ProductDataManager` [STABLE]
- 保持单例模式。
- `m_records` 将作为“入库流转”的核心账本。
- 商品库存更新逻辑：`record.quantity` 叠加到 `ProductInfo.stock`（仅当条码匹配时）。

## 4. UI/UX 进阶方案 (ui-ux-pro-max)
- **搜索优化**：支持拼音首字母检索商品（例如输入 `hjjx` 匹配 `皇家机械`）。
- **视觉反馈**：
    - 已上架商品：预览卡片背景为浅蓝色（系统信任色）。
    - 未上架商品：预览卡片背景为浅黄色（警告色），提示“档案缺失”。
- **校验逻辑**：
    - 数量必须 > 0。
    - 生产日期不可晚于当前系统时间。

## 5. 验证计划
- **自动化测试**：
    - 模拟录入已上架商品条码，验证是否自动填充。
    - 模拟录入新条码，验证入库后“库存看板”是否**未**出现该商品。
- **手动验证**：
    - 检查侧边栏新按钮的点击跳转。
    - 验证入库成功后，原有商品的库存数值是否实时递增。
